# v4.27 — trailer_type (39) → body_type (8) mapping  **[DRAFT — pending BA sign-off]**

The BOM rules engine keys on the **8 DDM body types**. But `icb_costings.calculations.trailer_type_id`
points at the **39 granular catalogue trailer_types** (size × variant). The BOM-on-accept hook
(`_build_jobspec_from_job`, WO §3.5) must therefore derive a body_type from the calc's trailer_type.

**Live surface today:** only **2** of the 39 are actually used by any calculation —
`4.9 & UP CHILLER AND 2.5 WIDE` (#27, 5 calcs → Chiller) and `4.9 & UP FREEZER BODY 2` (#21, 2 calcs →
Freezer). The rest of the map is future-proofing (and matters once more body types are quoted).

## Implementation: keyword matcher (not a frozen id list)
Match on the **uppercased name** in this precedence order — resilient to new size variants and to the
`[deleted-N]` retired entries (which keep their semantic name):

1. contains `EXPLOSIVE` → **Explosive** (structural-only — no geometry rules)
2. contains `FREEZER` → **Freezer**
3. contains `CHILLER` → **Chiller**
4. contains `ICECREAM` → **Icecream**
5. contains `DRY FREIGHT` / `DRYFREIGHT` → **Dryfreight**
6. contains `MEAT` → **Carcass**          *(BA prior: MEAT BODY → Carcass)*
7. contains `GRP` or `RHINORANGE` → **Insulated Trailer**   *(BA prior, “if insulated” — ⚑ verify)*
8. otherwise → **unmapped** → BOM persists `bom_status='incomplete'`, `metadata_json.body_type_unmapped=true`
