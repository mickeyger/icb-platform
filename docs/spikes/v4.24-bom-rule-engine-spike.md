## 3. Scaling assessment (the decision driver)

**Finding: one section ≠ one mechanism.** The single "Vacuum Materials" section is **47 BOM lines
across ~5 distinct calculation mechanisms**:

1. **Panel-count (foam)** — `ROUNDUP(dim/1220)` — *ported* (5 lines).
2. **Panel-count / area (plywood skins)** — `ROUNDUP(face_area/sheet_area)` — *ported* (4 lines).
3. **LVL beam/block counts** — different count rules (~4 lines) — not ported.
4. **GRP-sheet area (m²)** — material-selected, area-driven (~20 lines, mostly 0) — not ported.
5. **Resin/glue weight (kg)** — consumption ratios per m² of GRP (~14 lines) — not ported.

The qty side of **VACUUM ORDERS alone is 516 formula cells** (rows 1–30): ~94 cut-list cells, a
~32-cell spec-resolution block (`AF`/`AG`/`AH` ← `2026 COSTINGS`), and **5 local description-lookup
tables** (`H/L/N/P/R 2:66`), all body-type-branched. Porting the *panel-count* mechanism (9 functions)
required reading and resolving all of that.

### Side-by-side effort estimate for full-Module replication (Phase 3)

Assumptions: the Module has ~8–10 cost sections (Vacuum, Steel × {orders, fabrication, bending}, Prep,
GRP PS&Bill, Consumables, Extras, Fridge…), ~3–5 calc mechanisms each, and ~5–6 body types
(Freezer/Chiller/Dryfreight/Icecream/Insulated/…). Ranges, not point estimates.

| | **Hand-port (procedural)** | **Rules-table / DSL (data-driven)** |
|---|---|---|
| **Upfront** | ~0 (this spike's pattern) | **~15–25 dev-days** — a geometry-rule DSL/data model + evaluator, a materials/description→code data model, workbook-extraction tooling, the spec-resolution mapping |
| **Per section×mechanism** | ~1–2 dev-days bespoke code + per-material desc→code map + tests (this spike ≈ 2 days for 1 mechanism) | ~0.5–1 day **data entry/extraction** (no new code) |
| **Full Module** | **~60–100+ dev-days** of bespoke, branch-heavy code (~30–40 mechanism-ports × body-type variants), **high** ongoing maintenance | **~35–55 dev-days** total, then **low** maintenance (rules are data, auditable/editable by non-devs) |
| **Brittleness** | High — cross-sheet refs, volatile branches, description-template quirks baked into code | Contained — rules + maps are inspected data; the evaluator is small and tested once |
| **Crossover** | Cheaper only for 1–2 sections | **Wins decisively at full-Module scale** |

**The spike is good news for a rules-table:** the geometry rules proved **cleanly extractable and
deterministic** (9/9 exact), and every BOM part **resolves to the real SAP item master**
(`icb_sap.OITM`, 99% from the v4.23 pre-flight; 9/9 here). Both are prerequisites for a data-driven
engine. The blocker to hand-port isn't *correctness* — it's *volume × branching × per-material maps*.

## 4. Pricing assessment (§0.3 — was `U_LastPurchasePrice` right?)

**No — for replay fidelity.** `OITM.U_LastPurchasePrice` (loaded v4.23 from `04 - Inventory 2026.xlsx`,
a monthly SAP export per Simeon's TBC-6) is a **different snapshot** than the Costing Module's internal
**Table15 "Item Price."** Every line diverges (−12% … +18%), and the Module carries **0-price data gaps**
(GRP-POL-A-0158). The spike proved (test) that with Table15 prices the totals match exactly — so this is
purely a *source* question, not an engine error.

**Recommendation:** the production engine should price from **live SAP (OITM)** — it is *more current*
than the Module's stale internal snapshot, and the 0-price gaps are defects the new system fixes. Treat
the divergence as a **feature** (live pricing), and surface a "price vintage" indicator. `U_LastEvaluatedPrice`
is the alternative (moving-average valuation) if costing should reflect stock-on-hand value rather than
last purchase. Decide in the follow-on; do **not** replicate the workbook's stale internal prices.

## 5. Risks for follow-on WOs

- **Cross-sheet dependencies:** geometry reads `'New Prep'!AO8` (panel length 2440↔2630) and
  `'2026 COSTINGS'!D*` (resolved specs). Resolved to job-32735 values here; a rules-table must model
  these inputs explicitly.
- **Description-template brittleness:** descriptions are string-built from (thickness, material) with
  inconsistencies (`"24 DV"` vs `"32DV"`). We sidestepped it with a ported map; a rules-table needs a
  robust material→description→code data model (not string reconstruction).
- **Body-type branching:** `dryfreight`/`icecream`/`insulated`/Freezer paths permeate every formula.
  Each body type adds variant constants — multiplicative complexity for hand-port.
- **Per-section mechanisms:** GRP-area + resin-weight + LVL-count are *additional* engines per section.
- **Workbook data gaps:** 0-priced lines, `#REF!` cells observed elsewhere — the legacy file is not a
  clean source of truth.
- **Two-hop maps per section:** each section/material-family needs its own description→code map ported —
  scales with breadth.

## 6. Proposed follow-on WO chain

1. **v4.25 — Rules-table/DSL foundation.** Build the data-driven geometry-rule model + small evaluator
   + materials/description→code data model + workbook-extraction tooling. Re-implement **Vacuum × Freezer
   panel-count on it** to prove parity with this spike (same 9/9). *Decision gate: does the data-driven
   engine hold up?*
2. **v4.26 — Spec-resolution layer.** Port the DDM dropdown→spec resolution (147 dims) so the engine
   accepts raw job-card dropdowns, not pre-resolved specs. Closes the §3 spec-resolution boundary.
3. **v4.27 — Breadth + production wiring.** Add remaining Vacuum mechanisms (GRP-area, resin-weight,
   LVL) + a 2nd body type (Chiller) on the rules-table; settle the **price-source** decision (live SAP
   vs valuation); add **persistence + `production_jobs` integration** and the deferred
   `demand_lines.sap_code → icb_sap.OITM` FK (v4.23 carry-forward) once BOMs are persisted.

## 7. Recommendation

**PROCEED — but pivot from hand-port to a rules-table/DSL for Phase 3 §4.1.** The spike succeeded at
its real purpose: it proved (a) the Module's quantity rules are faithfully replicable (9/9 exact),
(b) the parts resolve cleanly to the v4.23 SAP-mock landing zone, and (c) hand-porting the *full*
Module does **not** scale (one section = 5 mechanisms / 516 cells / per-material maps; ~60–100+ dev-days
of brittle code vs ~35–55 for a data-driven engine that then grows by data, not code). Start the
follow-on chain at v4.25 (rules-table foundation), keep this spike's `geometry.py` as the reference
oracle for parity testing, and make the price-source a deliberate business decision (live SAP, not the
Module's stale internal prices). Per Q-Ph2D-05, this "pivot" recommendation **is** the spike's
successful outcome — not a failure.
