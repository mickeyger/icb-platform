# ADR 0030 — Chassis Records as the Master Record: the sole-editor chokepoint, optimistic lock, per-field audit (v4.36.5)

- Status: Accepted
- Date: 2026-06-28
- Work Order: v4.36.5 — Chassis Lifecycle Refactor (Phase 1.5; squash-merge ~1 Aug 2026)
- Builds on: ADR 0026 (chassis data integrity + the commitment-point / silent-deferral group-H lessons this extends to cross-phase scope), ADR 0023 (integrity invariants + the Tier-2 snapshot→dry-run→commit demo-reset discipline), ADR 0021 (chassis pipeline + the atomic job↔chassis link this edit door must not bypass), ADR 0016 (DB-keyed permissions + per-role journey testing), ADR 0015 (chassis lifecycle), ADR 0011 (`icb_test` not locally provisionable → DB/journey tests execute on CI).
- Parallel to ADR 0028 (CA1 v4.36c Kenny QC): shares the **alembic ship-order coordination** and the **audit name-snapshot** pattern; the two ADRs cross-reference the "which authz layer is load-bearing" question from opposite sides.

## Context

`chassis_records` already existed (ADR 0015/0021) but it was not yet *THE* master record. Chassis attributes were writable from more than one door — the Chassis-page Edit modal **and** Planning-Ack's `record_planning_ack`, which wrote **11 chassis fields directly** — with **no per-field audit, no optimistic lock, and no single enforcement point**. As ICB enters the Phase-1.5 parallel run (Simeon + the floor on the live MES while SAP is retired), "who changed the VIN / customer / make, and when" has to be answerable from the data, and two people editing the same chassis must not silently last-write-wins.

v4.36.5 promotes `chassis_records` to the master record along three axes: (1) the **Chassis page is the sole editor** of chassis attributes; (2) every attribute change is **audited per-field**; (3) concurrent edits are **optimistically locked**. The original WO framed this as hard 409-redirects on every read-only surface, per-row ownership, and a JSONB audit blob; **§3.0 discovery re-specced all three against the real code** (`docs/audit/v4_36_5_S3_0_chassis_lifecycle_refactor_discovery.md`) — the four-question reshape below. The sprint ran in parallel with CA1's v4.36c (`icb` + repo shared) and CA5's v4.37, so alembic ship-order and lane separation shaped it beyond the feature. Every decision was ratified at a §8 checkpoint; the **footnote ledger** records the code-grounded corrections that compounded along the way — several of which improved on the WO as written.

## Decisions

1. **One attribute-write chokepoint: `_apply_chassis_fields`.** All chassis-attribute writes funnel through a single service function (`services/chassis.py`) that, per field, applies the change *only on a real diff* and appends a `chassis_records_audit` row for the audited fields. It is the single point that turns "a field changed" into "the field is set + the change is trailed". Both the Chassis-page edit (`update_chassis`, `source="chassis_page"`) and the Planning-Ack propagation (`source="planning_ack"`) call it — so there is exactly one place where chassis attributes mutate, and exactly one place audit is emitted. The §3.0 BLOCKER-2 drift (Planning-Ack's 11 direct writes) was resolved by **routing `record_planning_ack` through this chokepoint** rather than forcing the chokepoint's shape onto the ack-as-written — behaviour-preserving (the existing ack journeys stayed green).

   **§3.8 hardening (post-§3.5-close BA review).** This claim was *aspirational* as first shipped: the BA's 7-angle review found four more write sites bypassing the chokepoint — `capture_vin`, the four lifecycle status-transition handlers (+ the VCL Body-Gap lift), the `create_chassis` placeholder-adoption, and the Pre-Job `_auto_create_chassis` sync — all writing audited fields directly. §3.8 routed every one through `_apply_chassis_fields` (lifecycle transitions with a per-event `source` — the **Option-1** unified-history decision, so a status change shows in the §3.4 viewer alongside attribute edits *and* keeps its `chassis_lifecycle_events` record), **moved `rec.updated_by` INTO the chokepoint** (so "write to chassis_records" and "go through the chokepoint" became synonymous and the direct-write habit that seeded the leaks is gone), and made the role-gate **fail-closed** (`actor_role` is now required — a caller that omits it is refused, not waved through). The claim is now *actual*.

2. **Production is read-only on chassis attributes via a SERVICE ROLE-GATE — not a permission, not an ownership column (Q1).** The WO's "production edits its own-assigned chassis" has **no backing column**: neither `chassis_records` nor `production_jobs` carries an assignee FK, and `'expected'` stubs/orphans have no job link at all — so "own" is unspecifiable without inventing schema. We dropped per-row ownership from the MVP entirely. Crucially, the enforcement is a **role-gate inside `update_chassis`** (`actor_role ∈ {admin, planner}`, else 403), NOT the `chassis.update` permission — because **`production` HOLDS `chassis.update`** (migration 0013, for VCL/DCL etc.). The permission alone would let production edit; the service role-gate is the **load-bearing** control. This is the mirror image of ADR 0028 decision 4, where the permission gate is load-bearing and there is no role-gate on top — when reusing either pattern, verify *which* layer actually blocks (`[[feedback-verify-permission-grants]]`).

3. **Planning-Ack stays INLINE through the chokepoint, not a hard redirect (Q2).** The ack is the one legitimate non-Chassis-page writer: the ETA capture **gates the Acknowledge button**, and the chassis is often a fresh `'expected'` stub at that moment, so bouncing the planner to the Chassis page mid-acknowledge would break the workflow. Instead the ack is an **authorized chokepoint caller** (`source="planning_ack"`) with the §3.9 attested fields (chassis type / VIN) locked read-only. "Sole editor" therefore means *sole interactive editor surface*; the ack is a sanctioned programmatic writer, audited like any other.

4. **Audit is a dedicated `chassis_records_audit` table, with `edited_by_name` as a write-time SNAPSHOT — not a JSONB blob, not a users-join (Q3/Q5).** The table clones the `ProductionJobAudit`/0023 shape (per-field `field_name` / `old_value` / `new_value` / `source` / `edited_by_user_id` / `edited_by_name` / `created_at`; `chassis_id` FK `CASCADE`; indexed on `(chassis_id)` + `(created_at)`) rather than a JSONB column on the hot chassis row (queryable, indexable, doesn't bloat the master row). The actor's **display name is denormalized at write time**: the snapshot is the read/display path and survives a user delete (the cross-schema `edited_by_user_id` FK is `SET NULL`); the FK is for traceability only. The WO's "join `users` on `edited_by_user_id`" was the wrong read path — audit is historical, the name *as it was* is the truth (`[[feedback-audit-denormalize-name-snapshot]]`; same principle as ADR 0028 decision 5's `category_name`). Raw machine values are stored for `source`; friendly labels are a frontend concern (decision 8).

5. **Structural lifecycle ops are audited too, through a separate `_audit_chassis` write point.** Merge / soft-delete / restore are **structural** events (they write `merged_into_id` / `deleted_at`), semantically distinct from attribute edits — so they do NOT pass through `_apply_chassis_fields` (an attribute differ). Instead `_apply_chassis_fields` and the structural ops both call a shared **`_audit_chassis` row-writer**, and `merge_chassis` / `soft_delete_chassis` / `restore_chassis` emit `source="merge" | "soft_delete" | "restore"`. Result: "who merged / deleted / restored this chassis?" is answerable from the *same* trail as "who changed the VIN?", which is exactly what the BA needs during parallel-run troubleshooting. (These ops are already `require_admin`; the audit is additive.)

6. **The optimistic-lock `version` is an etag — built as a backend gate in §3.1, wired live by the §3.3 caller.** `chassis_records.version` (server_default `0`) bumps once per real change; `update_chassis` 409s on a stale echo. §3.1 shipped the *gate* but **nothing was sending a version** — the lock was dead code shipping to prod (a cross-phase silent-deferral: a half-built feature with no caller reads as a working guard, `[[feedback-silent-deferral-as-defect]]` at engagement scale). §3.3 closed it: `ChassisRecordDetail` serializes `version`, and the Edit modal **echoes `version: rec.version` on every PATCH** (stale → "changed by someone else — reload"). The catch is that "the gate compiles" is not evidence "the lock works" — cross-phase wiring has to be verified end-to-end.

7. **Read-only surfaces get a centralized affordance, not a 409-redirect contract (Q4 scope trim).** §3.0 established that QC, Active-Jobs, and Pre-Job **do not write chassis attributes** (the WO's "QC updates chassis status" workflow was fictional; Pre-Job make/model/VIN are `PrejobCard` columns, not `chassis_records`). So there is no write to gate on those surfaces and **no `ApiError`/redirect contract was needed** — a large cut from the WO. The single shared form component (`ChassisFieldsForm`, used by both the Edit modal and the Planning-Ack panel) gains one optional **`editNotice` slot**: a non-editor render passes it to show a consistent "maintained on the Chassis page" banner with a deep-link; the sole editor omits it. Per-consumer wiring, **no shared provider** — each consumer supplies its own href.

8. **The §3.4 audit viewer: a lazy per-consumer hook, a collapsible section, a source-label map, gated on `chassis.update`.** `GET /api/chassis-records/{id}/audit` → most-recent-first, **router-capped at 200** (a long-lived chassis can't unbounded-render). `useChassisAudit` is a per-consumer hook that **lazy-fetches on first open** (no fetch for chassis views that never expand the section); the Chassis page renders one collapsible "Change history" table. `source` is rendered humanly via a frontend map (`chassis_page`→"Chassis edit", `merge`→"Merge", `soft_delete`→"Soft-deleted", …) — the **actual** source values, not the WO's illustrative `"attribute_edit"`. Authz is `require_permission("chassis.update")`: "if you can edit chassis, you can review who changed it." Because `production` holds that permission (decision 2), the viewer is intentionally **a bit broader than the edit audience** (admin/planner/production view; sales/workshop are refused) — a deliberate "more open" call: production is a legitimate forensic consumer during the parallel run, and it is read-only data.

9. **Alembic `0029` took its number by SHIP order, and the post-#57 rebase used MERGE, not rebase.** Per the ship-order rule (ADR 0028 §0.18 — the lane that ships *first* holds the lower number), v4.36.5's migration is `0029` chaining off CA1's v4.36c `0028`. Because CA1's `0028` was not yet on main while §3.1 was built, `0029` shipped with a `down_revision="0027"` **placeholder** + a DO-NOT-MERGE banner, to be re-pointed when v4.36c landed. (PR #56 was then merged early in error; CA1 re-pointed `0029→0028` on their branch, so post-#57 main carries the corrected chain `0027→0028→0029`.) The branch was reconciled by **`git merge origin/main`, not rebase** — §3.1's content had reached main via the #56 *squash*, so a rebase would replay already-merged commits and conflict; merge 3-wayed cleanly (`[[feedback-multi-ca-rebase-coordination]]`). The `test_smoke` table-count assertion rises **in the migration commit** (37 on the branch; reconciled to **40** at the rebase: 36 base + 3 v4.36c QC + 1 v4.36.5 audit), per the corrected §0.16 (CI builds the test DB via `alembic upgrade head`, so an early/model-introduction bump turns CI red).

10. **Tests execute on CI (`icb_test`); the role matrix is API-level, the edit→audit flow is a journey; live-verify is the three-gate reseed.** The hard db-guard refuses local non-`_test` DBs (ADR 0011), so DB-backed tests run on CI. The **role matrix** (`test_chassis_role_gate_api`) asserts the grants × edit/audit via `dependency_overrides[require_user]` — and proves the role-gate is load-bearing by showing `production` (a `chassis.update` holder) is still 403'd on a PATCH. The **UI loop** (`test_chassis_audit_trail_journey`) drives edit → version-echo → audit-row-render through the browser. Live-verify on shared `icb` followed the ADR 0023 Tier-2 discipline strictly: **snapshot (pg_dump, sha-logged) → clean dry-run → commit → confirm** (single `0029` head, 40 tables, role-gate live, health clean, the 13 visual-integrity flags preserved), gated on a CA1+CA5+admin idle-confirm (`[[live-verify-reseed-discipline]]`).

## Consequences

- **One editor, one write path, one trail.** Chassis attributes mutate in exactly one function; every change — attribute or structural — is answerable from `chassis_records_audit` alone ("who changed what, when, from where"), with the editor name immutable against later user deletes.
- **Concurrent edits are safe**, but only because §3.3 wired the etag the §3.1 gate was waiting for — a reminder that a guard without a caller is dormant, not done.
- **The MVP shrank, deliberately**: no ownership column, no 409-redirect contract, no JSONB — each cut traced to a §3.0 code-grounded finding, not a deadline. What shipped is load-bearing; what was cut was unspecifiable or unneeded.
- **Authz is two independent layers** (permission + service role-gate) whose reaches differ — `production` can view the audit but not edit attributes — a contrast worth holding next to ADR 0028's single-layer QC gate.
- **The demo dataset is reproducible**: the canonical `icb` state is `seed_v4_36c_demo_reset` (inherited from CA1 — v4.36.5 added no demo-reset script; the audit table is populated by representative chokepoint edits at verify time) behind the three-gate Tier-2 discipline.

## Footnote ledger — the checkpoint catches

- **§3.0 BLOCKER-1 (ownership):** "production edits its own chassis" had no backing column anywhere → per-row ownership dropped; production made flatly read-only via the service role-gate.
- **§3.0 BLOCKER-2 (gate location):** enforce at the `update_chassis` service chokepoint, not a router 409 — Pre-Job submit and Planning-Ack bypass the chassis router; `record_planning_ack`'s 11 direct writes were routed through `_apply_chassis_fields`.
- **§3.3 dormant 409:** §3.1 built the optimistic-lock gate but no caller sent `version` — the lock was shipping inert; §3.3 wired the version-echo and the lock went live (`[[feedback-silent-deferral-as-defect]]` at cross-phase scope).
- **§3.4 audit design:** `edited_by_name` snapshot (not a users-join), and the human source-label map over the **real** `source` values — both improved on the WO-as-written (`[[feedback-audit-denormalize-name-snapshot]]`).
- **§3.5 permission-scope correction:** `chassis.update` is granted to admin/planner **AND production** (0013) — broader than "the audience that edits"; this both proves the role-gate load-bearing and sets the audit-view breadth (`[[feedback-verify-permission-grants]]`).
- **§3.6 reseed:** canonical `icb` verified with the three-gate pattern (snapshot rollback 828 KB sha-logged, dry-run clean, commit + post-state proven). A Windows-local cp1252 crash on the env-guard's `⚠` banner (CI/Linux is UTF-8) was worked around with `PYTHONUTF8=1` — a v4.36c-script polish note, not a v4.36.5 defect.
- **§3.5-close review → §3.8:** the BA's 7-angle finder/verifier on the squash diff found 8 confirmed issues — `_apply_chassis_fields` was not yet THE single write point (capture_vin, 4 status transitions + the body_gap lift, create-adoption, Pre-Job sync all bypassed); the role-gate was fail-permissive (opt-in); the soft_delete audit stored the reason string under `field="deleted_at"` while the column got a timestamp; and `actor_id` was dropped on the ack + retrofit. §3.8 closed all of them, made the gate fail-closed, fixed the audit shape (real timestamp + a separate `deletion_reason` row), and added **7 assertion-level per-site tests** — the §3.5 journey edited only Notes, which masked the bypasses (the verification-blind-spot lesson). The conditional-ratify discipline (`[[feedback-verify-ci-green-before-ratify]]`) kept every phase-close provisional precisely so this deeper pass could land before the ~1 Aug merge.
- **§3.8 close-verify → §3.9:** the BA's verification sweep on the §3.8 diff found 3 MORE status-transition leaks the location-based passes had missed — `qc.signoff` (dispatch), `_release_auto_created_chassis` (card-unlink orphaning), and `reconcile_anchorless_chassis` (the system orphaning sweep). §3.9 routed all three (`source=qc_passed|unlink_card|reconcile_orphaned`), so **every** chassis status transition now flows through the chokepoint — decision 1's "status changes show in the §3.4 viewer" is now fully true (grep-proven: zero remaining chassis-status direct writes). The deeper lesson (`[[feedback-sweep-by-pattern-not-named-sites]]`): a chokepoint claim is a COMPLETENESS claim and must be verified by a MECHANISM sweep — grep every direct write to a guarded field and classify each — not a named-site walk. Captured as the completeness artifact `docs/audit/v4_36_5_S3_9_chokepoint_completeness_sweep.md` (the §3.0-style discovery for §3.9), which also flags 2 structural-op side-effects (the soft_delete notes-marker, the merge winner job_number) as audited-at-the-event-level exclusions pending a route-or-keep call.
