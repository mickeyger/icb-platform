# v4.27 — Per-body-type BOM oracle snapshot guide

**Purpose.** Produce a numeric BOM "oracle" for each non-Freezer body type, so the rules engine
can be validated **line-by-line** against the workbook (not just structurally). Freezer already has
one (job 32735). This guide is for **Nadie** (the Excel part) + the code agent (the extraction part).

## Why a human has to do this
`COSTING MODULE 2026.xlsx` is a **single-job, single-body-type live calculator** — one Body Type
dropdown re-resolves the *entire* workbook, and the `2026 BOM` sheet only ever shows the body type
currently selected. The code agent's tooling (`openpyxl`) can only read **cached** cell values; it
cannot make Excel recalculate after changing a dropdown. So each body type's numbers must be produced
**in Excel, by a person**, then saved. One saved file per body type = one oracle.

## Nadie — the steps (≈5 min per body type)
1. Open a **copy** of `COSTING MODULE 2026.xlsx` (never edit the master).
2. On the **`2026 COSTINGS`** sheet, pick a real, representative job for the target body type
   (ideally a recent quote you trust) and enter its job/quote number + customer.
3. Set the **Body Type** dropdown to the target (Carcass / Chiller / Dryfreight / Insulated Trailer /
   Hatchery / Icecream).
4. Set **all the spec dropdowns** to match that job: each panel's material + thickness
   (roof, sides, floor, front, rear, partition), the dimensions (length / width / height), and the
   frame/reveal inputs — exactly as you would when quoting it for real.
5. Let Excel **finish recalculating** (give it a moment; the `2026 BOM` sheet updates).
6. **Sanity-check** the `2026 BOM` sheet — the *Vacuum Materials* section and the grand total should
   look right for that job. If something looks off, fix the inputs before saving.
7. **Save As** (keep it `.xlsx`): `<jobnumber> - <BodyType> - oracle.xlsx`
   — e.g. `33012 - Chiller - oracle.xlsx`.
8. Send the file(s) to Michael / the code agent. That's it — no further Excel work.

## What the code agent does with each file (no action needed from Nadie)
- Runs `backend/app/spikes/v4_24/extract_fixture.py` against the saved workbook to snapshot the input
  spec + the `2026 BOM` *Vacuum Materials* lines into a JSON fixture (same shape as the committed
  `backend/tests/spikes/fixtures/job_32735_vacuum.json`).
- Adds a **line-by-line parity test** for that body type — SAP code + quantity per line
  (tolerance ≤ 0.001 on qty, ≤ 0.01 ZAR on the grand total) — upgrading it from structural-only.

## Reference — the `2026 BOM` oracle layout (for transparency; the agent handles this)
| Col | A | B | C | D | E |
|---|---|---|---|---|---|
| | Material Description | **Quantity** | **SAP Item Code** | Unit Price | Total Price |

Section header rows carry the section name in A and the section total in D; the *Vacuum Materials*
section header is on row 5, and the grand total ("TOTAL COST OF SALE") is on row 2. Line items start
at row 7.

## How this lands in the codebase
Until a body type's oracle arrives, its rules ship **structural-only** (resolves + generates a
non-empty BOM, but no numeric workbook check). As each snapshot comes in, the agent fixtures it and
flips that body type to **line-by-line** parity — folded into **v4.27.1** (or mid-WO as files arrive).
Nothing else in v4.27 is blocked on these.
