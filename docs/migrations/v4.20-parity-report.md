## Method

Load order (all scripts in `backend/migrations/pgloader/`, run as the table owner `icb_app`):

1. **`01_truncate_preload.sql`** ‚Äî clear the mock seed (43 `icb_costings` tables + all `icb_mes` tables), `RESTART IDENTITY CASCADE`. Preserves `branches`, `alembic_version`, and the two skip-list tables.
2. **`01b_drop_fks.sql`** ‚Äî capture all 53 `icb_costings` FK definitions into `_fk_backup`, then drop them. (Lets the load insert source orphans without RI-trigger errors ‚Äî `icb_app` is non-superuser and cannot `disable triggers`, which is why pgloader's own `disable triggers` was removed.)
3. **`grp_costings.load`** ‚Äî pgloader data-only load, `grp_costings` ‚Üí `icb_costings` (schema rename). Skip-list: `user_sessions`, `help_request_log`.
4. **`02_post_load.sql`** ‚Äî `branch_id = JHB` backfill on every branch-scoped base table; setval sweep over all owned sequences (belt-and-suspenders to pgloader's own reset).
5. **`03_readd_fks.sql`** ‚Äî re-add the 53 FKs (`VALID` where data satisfies them; `NOT VALID` fallback where source orphans exist).

**Skip-list rationale:** `user_sessions` (ephemeral login state) and `help_request_log` (operational log) are not catalogue data; they keep the target's own (empty) state.

## Summary

- **43 / 43 data tables at exact row-count parity.**
- **2 deliberate skips** (skip-list, expected divergence): `user_sessions`, `help_request_log`.
- **2 PG-only infra tables preserved** (not present in source): `branches`, `alembic_version`.
- **Independent cross-check:** Œ£ source rows excluding the two skip-list tables = **28,580**, which is *exactly* pgloader's reported import total (`‚úì 28580  3.2 MB  1.856s`, 0 errors). Every source catalogue row landed.

## Source data-quality drift (orphans)

**357 of 8,984 `bill_of_materials` rows (4.0%)** reference parent ids that do not exist in the source data ‚Äî a pre-existing referential-integrity gap in the faje UAT MySQL DB (MySQL's restored dump had FK enforcement relaxed, so these were never rejected). They are migrated as-is and isolated behind `NOT VALID` constraints. **No data was dropped or altered to make the load succeed.**

When faje cleans these up (or when WO v4.21+ reloads with corrected references), the constraints can be promoted with `ALTER TABLE ‚Ä¶ VALIDATE CONSTRAINT ‚Ä¶`. Until then they are visible and queryable but won't block costing reads.

## First-boot scaffolding delta (expected, non-catalogue)

The parity counts above are the **immediately-post-load** state. On the app's first boot against the
migrated DB, the idempotent `seed()` self-heals UI scaffolding only:

- `themes`: **3 ‚Üí 4** ‚Äî adds the inactive "Light" theme (`/static/css/theme-light.css`, `is_default=false`).
  faje's UAT export predates this theme; the active/default theme is unchanged.
- `admin_settings`: **no change** ‚Äî faje already has `nav_group_1/2/3`; all seed inserts are no-ops.
- `users`: **no change** ‚Äî seed is guarded (`if not any User`); faje's 5 users + hashes are untouched.

No catalogue table (materials, BOM, customers, calculations, trailer_types, ‚Ä¶) is touched by seed.

## Re-runnability

The whole sequence is captured by `backend/scripts/migrate_catalogue.ps1` (orchestrator) and the SQL scripts above. Parity can be re-checked any time with `parity_pg.sql` (PG side) and `parity_mysql.sh` (MySQL side, in WSL).
